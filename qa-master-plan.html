01 — Framework

METODOLOGÍA DE PRUEBAS

Seis capas de testing ordenadas por criticidad y momento de ejecución. Cada tipo tiene un rol específico en el ciclo de desarrollo del portal.

🔥
T-01 · SMOKE TEST
Prueba de Humo
Verificación rápida (≤5 min) de que el sistema arranca y sus funciones críticas responden. Es el primer filtro — si falla, se aborta el ciclo.
  • Portal carga sin errores (200 OK)
  • Login de usuario funciona
  • Supabase responde (<1s)
  • Cloudflare Worker activo
  • Assets críticos presentes
Cuándo: Al inicio de CADA sesión de QA, post-deploy automático y pre-merge a main.
🧠
T-02 · SANITY TEST
Prueba de Cordura
Valida que una funcionalidad específica recién modificada opera correctamente. Foco estrecho — no recorre todo el sistema, solo el módulo afectado.
  • Fix de bug confirmado resuelto
  • Nuevo feature básico funciona
  • Configuración de empresa activa
  • Módulo archivos.html post-fix
Cuándo: Después de cada hotfix, antes de pasar a regresión.
🔄
T-03 · E2E TESTING
Prueba de Extremo a Extremo
Simula flujos reales de usuario completos desde autenticación hasta resultado final. Detecta integraciones rotas entre módulos y el stack completo.
  • Login → Ver Leads → Crear Lead → Cerrar Lead
  • Upload → Carpeta → Descarga de doc
  • Admin → Stats → Reporte exportado
  • Cambio de empresa activa en navbar
Cuándo: Release candidates, viernes antes de deploy a producción.
👤
T-04 · UAT
User Acceptance Testing
Prueba final con usuarios reales del negocio. No prueba código — prueba si el sistema resuelve el problema de negocio como se definió. Nexus-Demo es el entorno ideal para UAT.
  • Flujo real de agente comercial (Jorge)
  • Revisión admin IBANOR (nibarra)
  • Validación de reportes por Alejandro
  • Aprobación formal antes de release
Cuándo: Pre-producción. Solo después de E2E aprobado. En Nexus-Demo.
🔁
T-05 · REGRESSION
Prueba de Regresión
Verifica que cambios nuevos no rompieron funcionalidades existentes. La suite qa-v8.html con sus 42 tests automatizados ES el regression suite del portal.
  • Aislamiento de empresas sin regresión
  • Sistema nervioso (9 tablas) OK
  • Auth y roles correctos
  • Worker Claude Proxy activo
Cuándo: Post-merge a main, post-hotfix crítico, semanal automatizado.
⚙️
T-06 · FUNCTIONAL
Prueba Funcional
Valida casos de uso específicos: valores válidos/inválidos, límites, permisos por rol. Orientada a requisitos funcionales documentados del portal.
  • Campos requeridos en formulario de leads
  • Restricciones por role (user vs admin)
  • Valores de fuente: referido, web, evento…
  • Filtros de empresa en todas las tablas
Cuándo: Al desarrollar features nuevos y en ciclos de QA completos.
02 — Documentación

PROTOCOLO DE DOCUMENTACIÓN

Los cinco entregables que deben existir para cada ciclo de QA formal. Son obligatorios en releases mayores; opcionales pero recomendados en hotfixes.

DOC-01
📋 Plan de Pruebas
Define el alcance, objetivos, recursos y cronograma del ciclo. Es el contrato del QA — qué se prueba, quién prueba y cuándo se considera "aprobado".
Scope:Módulos afectados por el release
Ambiente:Nexus-Demo (QA) → Prod
Criterio exit:≥95% pass, 0 blockers
Owner:acanales (super_admin)
DOC-02
🧪 Casos de Prueba
Los tests específicos a ejecutar: ID, descripción, precondiciones, pasos, resultado esperado, resultado real y estatus. La qa-v8.html representa 42 casos automatizados.
ID:ISO-001, SNS-003…
Prioridad:Critical / High / Medium / Low
Tipo:Automated / Manual
Empresa:IBANOR / MTX / Nexus / All
DOC-03
🗺️ Matriz de Trazabilidad
Conecta cada caso de prueba con el requisito funcional que valida. Asegura cobertura completa — cada feature tiene al menos un test, y cada test tiene una razón de negocio.
Formato:Feature → Test ID → Status
Cobertura mínima:100% features críticos
Herramienta:qa-v8.html SUITES map
DOC-04
🐛 Reporte de Bugs
Registro formal de cada defecto encontrado: ID, severidad, pasos para reproducir, empresa afectada, screenshot y estado de resolución.
Severidades:Blocker · Critical · High · Med · Low
SLA Blocker:Fix en <2h, re-test inmediato
SLA Critical:Fix en <24h
Canal:GitHub Issues + .github/workflows
DOC-05
📊 Informe Final de QA
Resumen ejecutivo del ciclo: pass rate, bugs encontrados/resueltos, cobertura, tiempo de ejecución y recomendación de go/no-go para producción.
Go criteria:Pass rate ≥95%, 0 blockers
Destinatarios:acanales + team leads
Fuente de verdad:qa-v8.html result JSON

04 — Core Security

⚡ qa-smoke.yml — Smoke Test Automático on: push, pull_request
01HTTP check — portal carga con 200 OK en <2s
02Supabase ping — endpoint REST responde, anon key válida
03Worker check — claude-proxy.acanales-7d4.workers.dev responde
04Assets críticos — app.js, config.js, fonts.css presentes
05Resultado — si falla cualquier paso, el PR queda bloqueado
🔁 qa-regression.yml — Regresión Semanal + Pre-release on: schedule (weekly) + workflow_dispatch
01Dispara qa-v8.html — corre las 9 suites (42 tests) vía Puppeteer headless
02Extrae JSON de resultados — pass/fail/skip por test con detail
03Verifica ISO suite completa — todos los tests de aislamiento deben pasar (0 tolerancia)
04Genera artifact — qa-report-YYYY-MM-DD.json subido al workflow
05Notificación — si pass rate <95%, abre GitHub Issue automáticamente
🗓️ Calendario QA — Ejecución Periódica schedule
DIARIOSmoke automático — uptime-monitor.yml (ya existe). Añadir check de Supabase + Worker.
SEMANALRegression suite completo — qa-v8 + reporte en GitHub Actions artifacts.
POR RELEASEE2E completo + UAT en Nexus-Demo — manual con usuarios jorge/nibarra antes de push a prod.

# qa-smoke.yml — estructura base para implementar
name: QA Smoke Test on: push: branches: [main] pull_request: jobs: smoke: runs-on: ubuntu-latest steps: - name: Check portal HTTP run: | STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://metatronixleads.tech) [ "$STATUS" = "200" ] || (echo "SMOKE FAIL: portal returned $STATUS" && exit 1) - name: Check Supabase REST run: | curl -sf -H "apikey: ${{ secrets.SUPABASE_ANON_KEY }}" "https://hodrfonbpmqulkyzrzpq.supabase.co/rest/v1/profiles?select=id&limit=1" || exit 1 - name: Check Cloudflare Worker run: | curl -sf https://claude-proxy.acanales-7d4.workers.dev/health || exit 1

07 — Ejecución

ROADMAP DE IMPLEMENTACIÓN

Plan de 4 sprints para pasar del estado actual (qa-v8.html manual) al QA framework completo con CI/CD integrado.

COMPLETADO — Sprint 0
✅ Base de QA — qa-v8.html operacional
42 tests automatizados en 9 suites. Aislamiento de empresas cubierto. Smoke, ISO, SNS, AUTH, WORKER, ROLES suites activas. Fix de company_id slug, RLS, y cross-company leakage aplicados.
EN CURSO — Sprint 1 (esta semana)
🟡 Smoke Test como GitHub Action automático
Crear qa-smoke.yml en .github/workflows. Corre en cada push a main y cada PR. Verifica portal 200 OK, Supabase REST, Cloudflare Worker y assets críticos. Bloquea PR si falla.
PRÓXIMO — Sprint 2
🔵 Regression semanal automatizado
Crear qa-regression.yml con schedule semanal. Usa Puppeteer/Playwright headless para correr qa-v8.html y extraer resultados a JSON. Sube artifact + abre Issue si pass rate <95%.
Sprint 3
🔵 E2E flows documentados + UAT checklist
Documentar los 5 flujos E2E críticos del portal. Crear checklist interactivo de UAT en Nexus-Demo para jorge/nibarra. Formulario de sign-off pre-producción.
Sprint 4
🔵 Dashboard de QA en tiempo real
Página /qa-dashboard.html que muestra el último resultado de qa-v8.html, histórico de pass rate, bugs abiertos y status de CI. Accesible para super_admin desde el sidebar.
